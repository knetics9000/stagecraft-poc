Core Principles

Internalize these before starting any phase. They are the load-bearing ideas — every phase decision flows from them.

PRINCIPLE 01
Prompt > Code
80% of the product's value lives in the prompt that tells the LLM how to reason. Wrapper code is the easy 20% — AI coding assistants write it for you. Spend your time on the prompt.
PRINCIPLE 02
Faisal in the loop
You cannot define "what stage is this deal at" alone. That's domain expertise Faisal has and you don't. Prompt iteration is a collaboration, not a solo build. Block his calendar weekly.
PRINCIPLE 03
Test set before prompt
Build 25–30 example meeting notes with "correct" answers labeled before writing the prompt. This is your ground truth. Without it, you're flying blind on every iteration.
PRINCIPLE 04
No scope creep
Only the stage classifier. No contacts database, pipeline view, dashboards, integrations, or auth. Every "wouldn't it be cool if" is a delay that kills the POC. Discipline beats ambition.
PRINCIPLE 05
Demo before optimize
Get something Faisal can show to sales leaders within 6–8 weeks. Polish comes after validation, not before. A rough demo with real customer reaction beats a perfect demo with no audience.
PRINCIPLE 06
Measurable wins only
Every phase has a specific deliverable. If you can't point to the deliverable, the phase isn't done. "We made progress" is not a deliverable.

PHASE 01
Build the Test Set
Days 3–5
Goal25–30 realistic meeting note samples, each labeled with the "correct" answer.

This is mostly Faisal's work. You facilitate, he provides the domain content.

Steps

  1. Pick the vertical Faisal commits to at the partner meeting (e.g., biotech BD)
  2. Define 5–7 sales stages with criteria, using MEDDIC as the default framework
  3. Faisal drafts 25–30 sample meeting notes (100–300 words each), spanning early-stage, deep negotiation, stalled, ambiguous
  4. For each note, Faisal writes the answer key: which stage, why, what's missing, what next action
  5. Save as JSON or markdown in test_data/

Why this matters

Without ground truth, you can't tell if a prompt is good or bad. With it, every iteration is measurable.

Done whentest_data/sample_meetings.json exists with 25–30 labeled examples.
PHASE 02
Prompt Engineering
Weeks 2–3
GoalA prompt that returns the correct structured stage judgment 80%+ of the time on the test set.

This is the heart of the project. Spend more time here than anywhere else. Most days of these two weeks should be iterating on the prompt with Faisal at your side or on call.

Steps

  1. Write the initial system prompt with: stage definitions, framework criteria (MEDDIC's 6 elements), output structure, reasoning instructions, 1–2 worked examples (few-shot prompting)
  2. Write an evaluation script that: loads test set → calls DeepSeek for each sample → saves response → compares to answer key → reports accuracy
  3. Run, review failures with Faisal, adjust prompt, re-run. Repeat.
  4. Lock when you hit 80%+ accuracy AND Faisal agrees reasoning quality matches "what I'd say in a pipeline review"

Common failures and fixes

  • AI hallucinates evidence not in notes → add "Only cite specific phrases from the notes. Do not invent details."
  • AI is too confident → add "If 2+ MEDDIC criteria are missing, confidence cannot exceed 60."
  • AI ignores the framework → restructure prompt: framework eval first, stage assignment second
  • Output structure inconsistent → use DeepSeek's response_format={"type": "json_object"} with a schema in the prompt

What to learn

  • How LLMs follow structured prompts (Anthropic's prompt engineering guide applies)
  • Few-shot prompting (examples inside the prompt)
  • Structured output with JSON mode
  • Writing evaluation scripts in Python with the openai library (DeepSeek is OpenAI-compatible)
Done whenprompts/v1.0_locked.md exists + evaluation script proves 80%+ accuracy on the test set.
PHASE 03
Build the Core API Function
Week 4
GoalA single function that takes meeting notes + company config, returns the structured judgment. No UI yet.

Steps

  1. Refactor the Phase 2 script into a clean reusable function. Inputs: notes string, config object. Output: structured JSON.
  2. Wrap as a serverless function on Vercel (Next.js or plain Node.js handler)
  3. Test the endpoint with curl or Postman
  4. Add basic error handling: malformed JSON, empty notes, rate limits, bad API key — don't over-engineer, just don't crash

What to learn

  • What a serverless function is (cloud function called via HTTP — Vercel makes this easy)
  • How to deploy to Vercel (push to GitHub, auto-deploy — about 30 min to learn)
  • Testing API endpoints with Postman or curl
Done whenA live URL like https://stagecraft-poc.vercel.app/api/classify returns a structured judgment when called.
PHASE 04
Build the UI
Weeks 5–6
GoalA simple, demo-ready web app that wraps the API function. Lovable or Bolt earns its keep here.

Don't write React from scratch. Describe what you want and let the AI build it.

Steps

  1. Open Lovable and describe the app in plain English (sample prompt below)
  2. Iterate: AI generates code → you preview → say "make output card bigger, add copy button" → repeat
  3. Hard-code 2–3 example companies in the setup screen so demos work without configuration
  4. Hard-code 3–5 example meeting notes the rep can load with one click
  5. Connect UI to your Vercel API endpoint from Phase 3
  6. Test end-to-end: pick company → load sample note → click Analyze → see AI response

Sample Lovable prompt to start with

"Build a sales CRM POC web app with three screens: (1) Setup — configure company stages and choose framework (MEDDIC/SPIN/BANT). (2) Input — large text box for meeting notes plus Analyze button. (3) Output — shows deal stage with confidence badge, plain-English reasoning, confirmed/missing framework criteria, recommended next action, risk flags. Clean professional dark mode. Analyze button calls the API at [your Vercel URL]."

Done whenFaisal can do a live screen-share demo without embarrassing either of you.
PHASE 05
Polish for Demo
Week 7
GoalThe POC is reliable enough for live demos to real prospects.

Steps

  1. Test the full flow 20 times in a row. Anything that breaks → fix.
  2. Add a graceful fallback if DeepSeek is slow or fails
  3. Ensure HTTPS works (Vercel does this automatically) and the UI works on mobile
  4. Take screenshots of the most impressive runs for the pitch deck
  5. Write a 2-minute demo script for Faisal
  6. Practice the demo with Faisal 5–10 times before he shows anyone real
Done whenFaisal can confidently put the demo URL in front of a sales leader.
PHASE 06
Validation Demos
Week 8 — Faisal-led
GoalGet the POC in front of 10 sales leaders and learn what they actually think.

This phase is Faisal's domain. You support by being on the demos to take notes, iterating on the prompt or UI based on consistent feedback, tracking what works in a simple spreadsheet, and updating mirofish quarterly to see if probability moves.

What you learn

  • What sales leaders actually care about (often different from what you assumed)
  • Which parts of the demo land and which fall flat
  • Whether the rep-narrative-input thesis holds against real customer questions
Done whenEither signed pilot agreements (best case) or specific learning about what needs to change before continuing.

What Could Go Wrong

Prompt accuracy plateaus at 60%
After 6 weeks of focused work. Means the AI can't reason about deals well enough. Action: Talk to Faisal. Decide: sharpen test set, simplify framework, or pivot product. Don't iterate blindly — the thesis itself is at risk.
Lovable/Bolt breaks on every change
Common with AI UI generators. Action: Switch tools (try the other one). Or have Cursor/Claude Code write React directly — slower but more controllable.
Faisal can't sit for prompt iteration
The project-killing risk. Action: Schedule in advance, treat as non-negotiable. If Faisal can't commit 3–5 hrs/week for 3 weeks, the project doesn't have a future.
DeepSeek breaks your integration
API changes happen. Action: Switch to another OpenAI-compatible provider (OpenRouter is easiest — routes to dozens of models through one API). Code shouldn't need major changes.
Hit Vercel free tier limits
Won't happen during POC — free tier is generous. Action: None needed until you have real traffic.
Demo is great but no one bites
Actually the best outcome of the POC — you learned truth cheaply. Either wedge is wrong, vertical is wrong, or pitch is wrong. Faisal pivots based on what they say no to.

When to Re-Run Mirofish

Quarterly Check-In

Re-run the simulation at each milestone

The probability number should rise as preconditions get met. If it isn't rising, something is wrong with execution and you need to inspect what.

  • After the partner meeting — capture new commitments, see probability shift
  • End of Phase 2 (prompt accuracy locked) — does technical validation move it?
  • End of Phase 6 (after 10 demos) — biggest moment; market feedback moves the number meaningfully
  • Anytime vertical, distribution strategy, or partnership terms change

Bottom Line

Eight weeks. Under $100. Real demo at the end.

The hard part isn't the code — it's the discipline to keep scope tight, keep Faisal at the table for prompt iteration, and ship something demoable instead of something perfect. If both of you commit to those three things at tomorrow's meeting, the build itself is achievable.
